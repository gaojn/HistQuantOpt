# HistQuantOpt 项目优化建议文档

## 一、总体建议

HistQuantOpt 已经具备 A 股指数增强和量化多头组合优化的基本闭环，包括数据读取、因子载入、组合优化、回测和报告输出。当前项目适合作为研究原型继续迭代，但如果要作为长期使用的投研框架，建议优先提升工程可靠性、测试覆盖、数据一致性和真实交易约束。

优化方向不建议先扩展更多策略，而应先把现有指数增强流程打磨稳定。

## 二、优先级最高的改进

### 1. 修复包管理配置

当前 `pyproject.toml` 配置为：

```toml
[tool.setuptools.packages.find]
where = ["src"]
```

但实际代码目录是：

```text
portfolio_optimizer/
```

项目没有 `src/` 目录，因此正式安装时可能找不到包。

建议二选一：

```toml
[tool.setuptools.packages.find]
where = ["."]
include = ["portfolio_optimizer*"]
```

或将代码迁移到标准结构：

```text
src/portfolio_optimizer/
```

### 2. 增加自动化测试

当前项目没有测试文件，风险较高。建议先补最小测试集：

- 优化器输出权重和是否为 1
- 个股权重上限是否生效
- 行业偏离约束是否生效
- 风格暴露约束是否生效
- 基准权重是否归一化
- 真实回测中涨停不能买、跌停不能卖、停牌不可交易
- 回测净值、换手率、现金统计是否合理

建议新增目录：

```text
tests/
├── test_index_enhance_optimizer.py
├── test_alpha_max_optimizer.py
├── test_benchmark_weights.py
├── test_realistic_backtester.py
└── test_data_panel.py
```

### 3. 修复真实回测现金统计

`RealisticBacktester` 中 `avg_cash_pct` 当前逻辑永远为 0：

```python
(port_values - port_values).clip(0).mean()
```

建议在每日循环中记录 `cash / nav_val`，最后计算平均现金比例。

### 4. 修复 Barra 因子日期过滤问题

`jy_barra.py` 中存在可疑逻辑：

```python
.filter(pl.col("date") == snapshot.industry.name or True)
```

`or True` 会导致过滤条件失效。建议明确传入 `target_date`，并只取目标日期截面。

## 三、策略逻辑优化建议

### 1. 完善涨跌停约束

当前优化器主要禁止停牌和次新股，但没有在优化阶段完整处理：

- 涨停：不能买入，只能持有或卖出
- 跌停：不能卖出，只能持有或买入
- 停牌：应保持上期权重，而不是简单置 0

建议优化器支持以下约束：

```text
SUSPENDED: w_i = w_prev_i
LIMIT_UP:  w_i <= w_prev_i
LIMIT_DOWN: w_i >= w_prev_i
NEW_LISTING/ST: w_i = 0
```

这会让优化结果与真实执行回测更加一致。

### 2. 引入真实风险模型

当前指数增强目标函数主要使用：

```text
alpha @ w - tracking_penalty * ||w - w_bm||²
```

这是 TE 的简化代理。建议后续加入 Barra 风险模型：

```text
active_risk = (w - w_bm)' Σ (w - w_bm)
Σ = B F B' + D
```

这样可以更真实地控制主动风险，而不是只控制权重距离。

### 3. 合成 Alpha 与真实 Alpha 分离

当前示例中使用合成 Alpha，且报告中绩效较高，容易被误读为真实策略效果。

建议明确区分：

```text
examples/synthetic/
examples/real_alpha/
```

并在报告中标注：

```text
该结果基于合成 Alpha，仅用于验证优化和回测流程，不代表真实可交易收益。
```

## 四、工程结构优化建议

### 1. 将 examples 中的核心逻辑下沉

当前 `examples/` 中包含大量正式流程逻辑，例如：

- 批量优化
- alpha 构造
- 候选池过滤
- 权重保存
- 回测执行

建议沉淀为正式模块：

```text
portfolio_optimizer/pipeline/
├── batch_optimize.py
├── backtest_pipeline.py
├── universe.py
└── config.py
```

`examples/` 只保留调用示例。

### 2. 增加统一配置文件

建议用 YAML 或 dataclass 管理策略参数：

```yaml
index: zz1000
start_date: 2023-06-01
end_date: 2026-05-22
rebalance_freq: 5
weight_upper: 0.01
min_constituent_ratio: 0.80
industry_active_bound: 0.07
style_active_bound: 0.50
max_turnover: 0.40
tracking_penalty: 10.0
```

避免参数散落在多个脚本中。

### 3. 增加 CLI

建议提供统一入口：

```bash
histquant optimize --config configs/zz1000.yaml
histquant backtest --weights output/zz1000_weights.parquet
histquant report --nav output/zz1000_nav.parquet
```

这样项目更容易复用和自动化。

## 五、建议实施顺序

### 第一阶段：稳定工程基础

1. 修复 `pyproject.toml`
2. 修复真实回测现金统计
3. 修复 Barra 日期过滤
4. 增加最小测试集

### 第二阶段：提高策略可信度

1. 完善停牌、涨停、跌停优化约束
2. 加入约束违约检查报告
3. 区分合成 Alpha 和真实 Alpha
4. 加入数据质量检查

### 第三阶段：框架化

1. 抽象 pipeline
2. 增加配置文件
3. 增加 CLI
4. 增加风险模型模块

## 六、结论

项目方向是正确的，当前最有价值的主线是指数增强。建议暂时不要扩展太多新功能，而是优先把现有闭环做扎实：能安装、能测试、约束真实、回测可信、结果可复现。

完成上述优化后，HistQuantOpt 可以从研究脚本升级为一个可长期维护的 A 股组合优化研究框架。
